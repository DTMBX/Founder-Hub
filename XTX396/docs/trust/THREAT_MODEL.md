# Threat Model — STRIDE Analysis

> B17-P2 | External Trust Layer | Classification: Security
>
> **Scope:** Evident/XTX396 platform — multi-tenant legal technology system
> handling evidence integrity, export generation, and operational automation.
>
> **Methodology:** STRIDE threat categorization with mitigations mapped to
> implemented B13–B16 controls.

## Assets

| Asset | Description | Sensitivity |
|---|---|---|
| Evidence Records | Tenant-owned data records subject to legal proceedings | Critical |
| Export Manifests | Watermarked, hashed export packages for law firm consumption | Critical |
| Audit Logs | Append-only event records of all system operations | High |
| API Keys | Tenant-scoped authentication credentials | High |
| Backup Bundles | Source-code and data backups with integrity manifests | High |
| Escrow Artifacts | Cryptographically sealed build artifacts with custody chain | High |
| Tenant Registry | Tenant identities, tiers, feature flags, rate limit profiles | Medium |
| SLO Configuration | Service level targets and alert thresholds | Low |
| Demo Mode Config | Public demo constraints and rate limits | Low |

## Trust Boundaries

| Boundary | Description |
|---|---|
| TB-1: Public Internet → Platform | Unauthenticated requests from public internet |
| TB-2: Demo Tenant → Platform Core | Public demo tenant with restricted capabilities |
| TB-3: Tenant A → Tenant B | Logical isolation between authenticated tenants |
| TB-4: Platform → Export Consumer | Exported manifests consumed by external systems |
| TB-5: Operator → Break-Glass | Emergency access requiring elevated authorization |
| TB-6: Platform → Backup Storage | Backup bundles written to local/offsite storage |

## Threat Analysis

### Spoofing

| Threat | Description | Mitigation | Control Reference |
|---|---|---|---|
| S-1 | Attacker presents forged or missing tenant context | Fail-closed tenant resolution; empty/unknown/suspended tenant IDs denied | TI-03 (`ops/tenancy/TenantContextMiddleware.ts`) |
| S-2 | Attacker reuses a revoked API key | Hash lookup rejects revoked keys; `revoked_key_attempt` audit logged | SH-04 (`ops/auth/ApiKeyManager.ts`) |
| S-3 | Attacker presents API key from another tenant | Key-to-tenant binding enforced; cross-tenant key usage denied and logged | SH-02 (`ops/auth/ApiKeyManager.ts`) |
| S-4 | Attacker impersonates break-glass operator | Two-person authorization required; single-person only with 15-min unreachable + mandatory review | BG-01, BG-02 (`governance/security/break_glass_playbook.md`) |

**Residual Risk:** API key format (`evk_` prefix + 64 hex) is guessable in
structure but not in value (cryptographic randomness). Key rotation cadence is
policy-driven, not automated.

### Tampering

| Threat | Description | Mitigation | Control Reference |
|---|---|---|---|
| T-1 | Attacker modifies export records after generation | SHA-256 integrity hash recomputed on verification; mismatch = rejection + tamper audit event | EX-03, EX-05 (`ops/export/ExportIntegrity.ts`) |
| T-2 | Attacker alters backup bundle contents | Per-file SHA-256 hashes in manifest; post-creation verification; monthly restore drill revalidation | BK-03, BK-06 (`ops/backup/BackupService.ts`) |
| T-3 | Attacker modifies audit log entries | Append-only storage; per-event SHA-256 payloadHash; modification prohibited by policy | AU-03, AU-05 (`ops/automation/audit/OpsAuditLogger.ts`) |
| T-4 | Attacker deletes protected source files | Pre-commit/pre-push hooks scan for deletions; >25% mass deletion blocked; override requires env var + documented reason | SP-01–SP-05 (`ops/backup/AntiDeletionGuard.ts`) |
| T-5 | Attacker tampers with escrow artifact | SHA-256 integrity verification on release; chain of custody metadata immutable | BK-09, BK-10 (`ops/backup/ArtifactEscrowService.ts`) |

**Residual Risk:** Append-only audit enforcement relies on application-layer
controls, not hardware-backed immutability. An operator with filesystem access
could theoretically modify log files. Hardware-backed immutability (e.g.,
write-once storage) is a future enhancement.

### Repudiation

| Threat | Description | Mitigation | Control Reference |
|---|---|---|---|
| R-1 | Operator denies performing break-glass action | Independent append-only logging of all emergency actions; Security Officer review | BG-03, BG-05 (`governance/security/break_glass_playbook.md`) |
| R-2 | Tenant denies export was generated by platform | Watermark with tenant ID, timestamp, system ID, batch ID, format version embedded in every export | EX-02 (`ops/export/ExportIntegrity.ts`) |
| R-3 | Actor denies API key creation or revocation | `key_created` and `key_revoked` events logged with keyId, tenantId, timestamp | SH-05 (`governance/security/api_key_policy.md`) |
| R-4 | Actor denies data access or modification | Structured audit events with UUID, ISO-8601 timestamp, actor, payloadHash for all operations | AU-01, AU-02 (`governance/ops/ops_audit_policy.md`) |

**Residual Risk:** Commit signing is recommended but not enforced. Non-signed
commits could be attributed incorrectly. GPG/SSH signing enforcement is a future
enhancement.

### Information Disclosure

| Threat | Description | Mitigation | Control Reference |
|---|---|---|---|
| I-1 | Cross-tenant data leakage via query | All queries scoped by tenant_id; cross-tenant access throws and logs | TI-02 (`ops/tenancy/TenantContextMiddleware.ts`) |
| I-2 | Cross-tenant data leakage via export | Cross-tenant records rejected at export creation time | EX-04 (`ops/export/ExportIntegrity.ts`) |
| I-3 | PII exposure in audit logs | Emails redacted to domain; phone to last 4 digits; IPs/passwords/keys never stored | AU-04 (`governance/ops/ops_audit_policy.md`) |
| I-4 | Demo user accesses excessive data | Output bounded: 100 rows, 5 MB exports, 10K-char previews | DM-01 (`ops/demo/DemoGuard.ts`) |
| I-5 | Secrets leak via configuration files | Pre-commit scanning; no secrets in shell configs; disk encryption required | SH-07, SH-09 (`governance/security/workstation_hardening_policy.md`) |
| I-6 | API key exposed after creation | Plain key returned once; only SHA-256 hash persisted | SH-01 (`ops/auth/ApiKeyManager.ts`) |

**Residual Risk:** Runtime memory may briefly hold decrypted secrets. Memory
isolation is delegated to the runtime environment and operating system.

### Denial of Service

| Threat | Description | Mitigation | Control Reference |
|---|---|---|---|
| D-1 | Volumetric request flooding | Per-IP rate limiting at 60 req/min; burst detection at 15 req/5s | AB-01, AB-02 (`ops/security/AbuseProtection.ts`) |
| D-2 | Repeated burst attacks | Auto soft-ban after 3 violations; 10-min duration; exponential backoff | AB-03, AB-04 (`ops/security/AbuseProtection.ts`) |
| D-3 | Demo mode resource exhaustion | Row limits, export size limits, feature restrictions, per-tenant rate limiting | DM-01–DM-03 (`ops/demo/DemoGuard.ts`) |
| D-4 | SLO degradation undetected | Tiered SLO monitoring; warning (4h) and critical (15-min) alert escalation | MO-01–MO-04 (`ops/monitoring/HealthMonitor.ts`) |

**Residual Risk:** Application-layer rate limiting does not protect against
network-layer DDoS. Infrastructure-level mitigation (CDN, WAF) is expected in
production deployment but is outside application scope.

### Elevation of Privilege

| Threat | Description | Mitigation | Control Reference |
|---|---|---|---|
| E-1 | API key scope escalation | Scope hierarchy enforced: read_only cannot perform export or admin actions | SH-03 (`ops/auth/ApiKeyManager.ts`) |
| E-2 | Demo tenant attempts restricted operations | DemoGuard blocks file upload, external APIs, billing, messaging | DM-02 (`ops/demo/DemoGuard.ts`) |
| E-3 | Tenant accesses another tenant's tools | Tool access controlled via tenant-scoped `allowedTools[]` | TI-06 (`ops/tenancy/TenantContextMiddleware.ts`) |
| E-4 | Unauthorized break-glass activation | Two-person rule; unauthorized use is a terminable offense; all events reviewed | BG-01, BG-05 (`governance/security/break_glass_playbook.md`) |

**Residual Risk:** RBAC is currently API-key scoped. Role-based access control
at the user level within a tenant is not yet implemented. This is a planned
enhancement for multi-user tenants.

## Residual Risk Summary

| Risk | Severity | Mitigation Status | Planned Enhancement |
|---|---|---|---|
| Application-layer audit immutability | Medium | Append-only policy enforced at application layer | Hardware-backed write-once storage |
| Commit signing not enforced | Low | Recommended in policy | Enforce GPG/SSH signing via branch protection |
| Network-layer DDoS | Medium | Application-layer rate limiting only | CDN/WAF integration |
| Runtime memory secret exposure | Low | Delegated to OS/runtime | Secure enclave consideration |
| Single-user tenant RBAC | Low | API key scopes enforce access | Per-user roles within tenants |
| Automated key rotation | Low | Policy-driven manual rotation | Automated rotation scheduler |
